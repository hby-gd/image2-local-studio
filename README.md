# Image 2 Local Studio

一个面向本地使用的图片工作台：你只需要有自己的 `API 地址` 和 `API Key`，就能直接在浏览器里完成文生图、图生图、局部改图、提示词整理、历史保存和导出。

这个仓库当前是：

- 单文件前端，核心就是 [`index.html`](./index.html)
- 无构建步骤，无依赖安装
- 默认纯前端直连接口
- 附带一个零依赖本地代理 [`server.js`](./server.js) 用来解决上游未开放 CORS 的情况

它不会把请求先发到作者服务器再转发。你填的地址和密钥，只会发往你自己配置的接口提供方，或者发到你本机运行的代理。

## 适合谁

适合这些场景：

- 已经买了中转或自有兼容接口
- 手里有 `API 地址` 和 `API Key`
- 不想自己写脚本、拼 JSON、调 SDK
- 希望把图片接口调用变成一个可视化页面

不适合这些场景：

- 需要平台代管账号、图片或任务队列
- 需要服务端多人共享
- 上游完全不兼容当前调用方式

## 当前功能

- 文生图
- 图生图
- 蒙版局部改图
- 提示词协作台
- 输出规格预设
- 本地历史记录
- 搜索、分页、批量选择
- 导出所选图片
- 从历史图继续做图生图参考
- 连接测试和图片模型测试

页面当前固定按 `gpt-image-2` 做图片生成；右侧提示词助手默认优先尝试 `gpt-5.5`，并可切换 / 降级到 `gpt-5.4`、`gpt-5.4-mini`。

## 快速开始

### 方式 1：直接打开单文件

适用于你的上游接口允许浏览器直连。

1. 直接用 Chromium 内核浏览器打开 [`index.html`](./index.html)
2. 推荐使用 Edge 或 Chrome
3. 点右上角 `连接设置`
4. 填入：
   - `API 地址`
   - `API Key`
5. 先执行：
   - `测试连接`
   - `测试图片模型`
6. 回到 `文生图` 或 `图生图 / 局部改图` 开始生成

### 方式 2：启动本地代理

适用于你的上游接口不开放浏览器跨域，页面直连会报 `Failed to fetch`、`浏览器直连失败` 或其他 CORS 相关错误。

先确认本机已安装 Node.js，然后在项目目录运行：

```bash
node server.js
```

默认监听：

```text
http://127.0.0.1:8787
```

建议直接在浏览器打开：

```text
http://127.0.0.1:8787
```

然后在页面里这样配置：

1. `API 地址`：继续填写你的真实上游地址
2. `API Key`：继续填写你的真实密钥
3. `访问模式`：选择 `本地代理`
4. `本地代理地址`：填写 `http://127.0.0.1:8787`

注意：

- `API 地址` 仍然是你的真实上游，不是 `127.0.0.1`
- `本地代理地址` 是单独配置项
- 代理只负责转发，不保存你的图片或密钥

## 连接设置说明

页面里主要会用到这些配置：

- `API 地址`
  - 可填根地址、`/v1`，也可直接填完整图片生成地址
- `图片生成地址（可选）`
  - 如果你的通道不是标准 `/v1/images/generations`，可以手动改成完整地址
- `API Key`
- `访问模式`
  - `浏览器直连`
  - `本地代理`
- `本地代理地址`
- `默认响应格式`
  - 推荐 `b64_json`
  - `url` 更轻，但部分通道会遇到跨域和二次下载问题

## 使用流程

### 文生图

1. 进入 `文生图`
2. 输入提示词
3. 选择尺寸、质量、响应格式、数量
4. 点击 `开始生成`

### 图生图 / 局部改图

1. 进入 `图生图 / 局部改图`
2. 上传参考图
3. 如需局部修改，再上传蒙版
4. 输入提示词后生成

说明：

- 上传参考图后会自动走图生图 / 编辑接口
- 上传蒙版但没有原图时，页面会阻止提交
- 历史区的图片可以一键放回去继续做参考图

### 提示词协作台

右侧助手会结合当前参数整理提示词，包括：

- 当前模式
- 尺寸
- 质量
- 响应格式
- 是否有参考图
- 是否有蒙版
- 最近使用的预设

它的定位是“整理提示词”，不是替代左侧真实的出图参数。也就是说，左侧的尺寸和质量才是最终提交给图片模型的配置。

## 历史与导出

历史记录默认保存在浏览器本地：

- 配置保存在 `localStorage`
- 图片历史保存在 `IndexedDB`

这意味着：

- 历史图不在项目目录里
- 你不会在系统图片文件夹里直接看到它们
- 重新打开同一个浏览器时，历史仍然可能存在

导出时支持两种方式：

- 普通下载
- Chromium 浏览器下使用文件夹选择器批量写入

如果你更在意可导出和可持久化，优先选择 `b64_json`；`url` 模式下某些结果只能预览，未必能稳定导出。

## 本地代理做了什么

[`server.js`](./server.js) 负责两件事：

1. 把当前目录作为静态站点服务出来
2. 提供 `/proxy` 转发上游请求，并补上浏览器可用的 CORS 头

当前代理还会处理这些情况：

- 保留连接复用
- 最多重试部分临时网络错误
- 跟随上游重定向
- 代理远程图片下载请求
- 提供 `GET /health` 健康检查

它是零依赖实现，只用 Node.js 内置模块，不需要 `npm install`。

## 已知边界

这些更像使用边界，不一定是项目 bug：

- 如果上游接口本身不兼容当前请求格式，页面无法替你适配
- 4K、自定义尺寸、360 全景等能力是否可用，取决于你的接口提供方是否支持
- `url` 响应格式在部分通道上可能只能预览，不能稳定落地导出
- 提示词助手依赖文本模型可用，图片模型能跑不代表助手一定能跑
- 浏览器直连模式下，跨域限制无法由前端自身绕过

## 常见问题

### 1. 页面能打开，但测试连接或生成失败

先检查：

- `API 地址` 是否正确
- `API Key` 是否可用
- 图片接口是否真的开通
- 是否被上游 CORS 拦截

### 2. 为什么建议优先用 `b64_json`

因为它更适合：

- 写入本地历史
- 二次打开预览
- 批量导出

而 `url` 模式虽然更轻，但更容易被远程下载限制或跨域限制影响。

### 3. 为什么历史图在页面里能看，在资源管理器里找不到

因为默认保存在浏览器的 `IndexedDB`，不是普通文件夹。要变成真实文件，需要手动导出。

### 4. 代理启动了，为什么还是不通

重点检查：

- 你打开的是否真的是 `http://127.0.0.1:8787`
- 页面里的 `本地代理地址` 是否也填成这个根地址
- `API 地址` 是否仍然是上游真实地址，而不是误填成本地地址

## 安全说明

这个项目的安全边界比较直接：

- 不内置第三方中转服务
- 不上传到作者服务器
- 不帮你保管图片
- 不帮你托管 API Key

但你仍然需要自己负责：

- 只在信任的机器上保存配置
- 不要在公共电脑长期保留密钥
- 自行评估你所使用的接口提供方是否可信
